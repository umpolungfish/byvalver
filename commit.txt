fix(tier1): Resolve compilation warnings and batch processing bad-char display bug

## Summary

Fixed all compilation warnings in newly implemented Tier 1 strategies to achieve
clean build (0 warnings, 0 errors). Also fixed a critical bug where batch processing
summary always showed "Bad characters: 0 configured" even when bad characters were
specified via --bad-chars or --profile flags.

## Problem Statement

### Compilation Warnings

The Tier 1 strategy implementation (polymorphic immediate construction, SETcc jump
elimination, register dependency chain optimization, and syscall number obfuscation)
compiled successfully but generated 5 compiler warnings:

1. **polymorphic_immediate_construction_strategies.c:**
   - Warning: 'estimate_encoding_size' defined but not used
   - Warning: 'try_lea_arithmetic' defined but not used

2. **setcc_jump_elimination_strategies.c:**
   - Warning: unused parameter 'insn' in can_handle_setcc_flag_accumulation
   - Warning: unused parameter 'insn' in can_handle_setcc_arithmetic_multiply

3. **syscall_number_obfuscation_strategies.c:**
   - Warning: unused variable 'reg_code' in generate_syscall_number_al_load

These warnings needed to be resolved before moving to the testing phase.

### Bad Character Display Bug

In batch processing mode, the summary always displayed:
```
Bad characters:    0 configured
```

Even when using `--bad-chars "00,0a,0d"` or `--profile http-newline`, the count
remained at 0. This occurred because the global bad character context was not
initialized before the batch statistics were populated, causing the stats to
copy from an uninitialized context.

## Changes Implemented

### 4. Bad Character Context Initialization (Critical Bug Fix)

**File**: src/main.c

**Changes**:
- Added `init_bad_char_context(config->bad_chars);` at line 495 (batch processing mode)
- Added `init_bad_char_context(config->bad_chars);` at line 665 (single-file mode)
- Removed redundant `init_bad_char_context()` call from inside `process_single_file()` (line 104)

**Rationale**:
- The batch statistics code (lines 500-507) copies bad char config from global context
- Previously, global context wasn't initialized until inside `process_single_file()`
- This was too late - batch stats had already copied from uninitialized memory
- Now context is initialized BEFORE batch stats setup, ensuring correct values

**Before (Incorrect)**:
```
===== BATCH PROCESSING SUMMARY =====
Total files:       7
Successfully processed: 7 (100.0%)
Failed:            0 (0.0%)

Bad characters:    0 configured    <- WRONG!
====================================
```

**After (Correct)**:
```
===== BATCH PROCESSING SUMMARY =====
Total files:       7
Successfully processed: 7 (100.0%)
Failed:            0 (0.0%)

Bad characters:    3 configured    <- CORRECT!
Configured set:    0x00, 0x0a, 0x0d
====================================
```

**Impact**: Users now see accurate bad character configuration in batch summaries,
enabling proper verification that the correct profile/bad-chars are being used.

### 2. SETcc Jump Elimination Strategies

- Added `__attribute__((unused))` to `insn` parameter in `can_handle_setcc_arithmetic_multiply()` (line 222)

**Rationale**:

- Comments indicate "TODO: Implement with instruction window analysis" and
  "TODO: Implement with multi-instruction analysis"
- Must maintain function signature for strategy registry compatibility
- Parameter will be used when multi-instruction lookahead is implemented

**Impact**: No functional changes to the two active SETcc strategies
(Simple SETcc Jump Elimination, SETcc to Conditional Move)

### 3. Syscall Number Obfuscation Strategies

**File**: src/syscall_number_obfuscation_strategies.c

**Changes**:
- Removed unused variable `reg_code` in `generate_syscall_number_al_load()` (line 103)
- Removed unused variable `dst_reg` from the same function

**Rationale**:
- Code comment indicated "EAX/RAX both code 0", meaning register encoding is fixed
- Variable was calculated but never used: `reg_code = (dst_reg == X86_REG_RAX) ? 0 : 0`
- Function always generates fixed XOR EAX, EAX and MOV AL, imm8 encoding
- No runtime register calculation needed for this strategy

**Impact**: No functional changes to syscall number obfuscation strategies
